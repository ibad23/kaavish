This chapter provides an overview of the project, starting with the core problems that motivated our work, the gap between UI design and code implementation, the difficulty of keeping a design system in sync at scale, and the absence of intelligent knowledge retrieval in CRM platforms. We then describe our proposed solution, outline the project plan including objectives, tasks, team roles, and timeline. 

\section{Problem Statement}

The following subsections describe the specific problems this project addresses.

\subsection{The Design-Implementation Gap}
One of the biggest challenges in product development today is the disconnect between design and implementation. Designers create visually appealing and user-friendly interfaces in Figma, the design tool of choice for UI/UX professionals, but translating these designs into production-ready code requires significant manual effort. Frontend developers must interpret design specifications like colors, spacing, typography, component states, and interactions and re-implement them in code, a process that is time-consuming, error-prone, and heavily dependent on communication between design and engineering teams. Even minor misinterpretations can result in visual inconsistencies that accumulate across an application, degrading user experience and requiring costly rework.

\subsection{Design System Scalability}
Design systems are the backbone of scalable SaaS products, bringing together reusable components, design guidelines, and patterns that speed up development and enhance the user experience. However, maintaining a unified design system across a growing codebase is a persistent challenge. As components evolve in Figma, their code counterparts often fall out of sync, designers update a component's spacing or color tokens, but the corresponding Angular code retains the old values. Existing B2B SaaS platforms, including SpurSol's Knoccs, struggle to maintain this synchronization at scale. Without an automated mechanism to detect and propagate design changes to code, teams rely on manual audits that are both incomplete and unsustainable as the component library grows.

\subsection{Lack of Intelligent Knowledge Retrieval}
Beyond the design-to-code challenge, modern CRM platforms like Knoccs handle vast amounts of client-specific data: emails, conversations, internal notes, tasks, attachments, and organizational documents. This knowledge is scattered across databases, storage systems, and individual team members, making it difficult for support agents to quickly retrieve accurate, contextual answers to client queries. Without an intelligent retrieval system, agents must manually search through documents and communication histories, leading to slower response times, inconsistent answers, and an overall decline in service quality. There is a clear need for a system that can consolidate this fragmented knowledge and provide grounded, evidence-backed responses in real time.

\subsection{The Knoccs Context}
SpurSol's Knoccs platform exemplifies all three of these challenges. Its development workflow follows a conventional pattern: designers create mockups in Figma, frontend developers manually translate these into Angular code, and backend developers implement server-side functionality independently. There is no automated pipeline connecting design to code, no mechanism for keeping components in sync with evolving designs, and no AI-powered system for intelligent client support. This project, therefore, targets a real production environment where these problems directly impact development velocity, code quality, and customer service efficiency.

\section{Proposed Solution}
This project, \textit{Design-to-Code with Knoccs} by SpurSol, addressed these challenges by creating a scalable design-to-code pipeline that translates design components into production-ready Angular code. The pipeline revamped the Knoccs design system and introduced a multi-agent AI layer capable of autonomously generating components of the codebase, along with a sync mechanism to keep generated code aligned with evolving Figma designs.
\\\\
In addition, the project leveraged this pipeline to implement new functionalities within Knoccs, specifically Tag and Macro Management features, and developed a RAG-based knowledge base for intelligent information retrieval. Together, these validated the system's scalability, performance, and real-world applicability within a production CRM environment.

\section{Project Plan}

\subsection{Project Objectives}

\begin{enumerate}
    \item \textbf{Component Library Development:} Develop a comprehensive component library in Figma to automate front-end development. This library allows developers to reuse pre-built Angular components instead of manually writing HTML and CSS, enabling faster and more consistent UI implementation.

    \item \textbf{Figma-to-Code Synchronization:} Build a sync pipeline that detects changes in Figma designs and propagates targeted updates to the corresponding Angular code, keeping the component library aligned with the evolving design system without requiring full regeneration.

    \item \textbf{System Integration and Feature Validation:} Convert Figma designs into Angular code components that integrate into the existing Knoccs codebase. Validate the generated components through the implementation of two features, Tag Management and Macro Management, to confirm functional accuracy and integration feasibility.

    \item \textbf{Knowledge Base (RAG) Development:} Using the developed component library, implement a Retrieval-Augmented Generation (RAG) system that builds a knowledge base supporting intelligent information retrieval within the Knoccs platform.
\end{enumerate}

\subsection{Project Tasks}

\subsubsection{Design-to-Code Pipeline}

\begin{itemize}
    \item Analyzed and understood the Knoccs design system and repository architecture.

    \item Revamped the Knoccs design system so that it provides rich, structured context suitable for use as an effective prompt for an LLM.

    \item Conducted a literature review on efficient methods to extract designs from Figma, research on different design systems that have implemented automated code generation, and comparative performance of LLMs for design-to-code generation.

    \item Integrated the design system with a Claude Code agent using the Figma MCP server, enabling real-time access to design metadata.

    \item Configured the Claude agent to accurately understand the context and generate the frontend in accordance with Knoccs, applying iterative prompt engineering across multiple instruction file versions.

    \item Generated missing components through Claude and validated them, completing the component library of 20 Angular components.

    \item Mapped all components in Storybook with all available states of those components.

    \item Built a Figma-to-Code sync pipeline that detects design changes and applies targeted updates to existing component code, achieving approximately 80-90\% automation.

    \item Developed a sync interface to make the pipeline accessible to designers, automating prompt generation and execution.
\end{itemize}

\section{Risks}

The following risks were identified during the project and influenced design decisions throughout development. These differ from the shortcomings and future work discussed in Chapter~6: risks are potential problems that could have disrupted the development process itself, whereas shortcomings are known limitations of the delivered system that remain to be addressed.

\begin{itemize}
    \item \textbf{Integration Challenges:} The generated Angular components risked not aligning with the existing Knoccs codebase structure. To mitigate this, the CLAUDE.md instruction file included detailed repository conventions and reference components, and all generated code underwent PR review by the SpurSol team.
    \item \textbf{Code Generation Inconsistency:} Claude could produce inconsistent or non-optimized Angular code across different generation sessions. This was addressed through iterative prompt engineering (nine versions of the instruction file) and strict human-in-the-loop governance at every phase gate.
    \item \textbf{RAG Retrieval Accuracy:} The Retrieval-Augmented Generation pipeline risked retrieving irrelevant or outdated information from customer documentation. Hybrid retrieval (BM25 + dense embeddings) combined with cross-encoder re-ranking was implemented to improve retrieval precision, and the system was evaluated against 52 ground truth queries using DeepEval.
    \item \textbf{Performance Bottlenecks:} Integration of Claude and RAG features risked increasing response time or server load during retrieval or generation. The RAG system was designed with efficient indexing (IVFFlat with pgvector) and company-scoped queries to keep retrieval fast, while the design-to-code pipeline operated as an offline process rather than a real-time service.
\end{itemize}
